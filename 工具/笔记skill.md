## 📌 Part 2：什么时候需要记笔记？### 什么时候应该加新内容？



当用户说以下内容时，AI 应该帮他记笔记：- 遇到了一个新坑，花了半小时才解决 → **记下来**

- 发现了一个更高效的方法 → **记下来**

✅ **一定要记：**- 某个分析技巧经常用到 → **记下来**

- "我遇到了一个问题，花了半小时才解决" — 这是典型的坑点- 其他时候 → 可以先不加，以后有机会再说

- "之前我不知道这样可以做" — 新方法、新技巧

- "这个步骤我经常用到" — 高频操作### 把新内容加到哪里？

- "记一下，别下次又忘了" — 用户主动要求

- "我想记下这个思路/方法" — 用户的明确需求| 新增内容 | 放在哪个文件 |

|---------|-----------|

❌ **不需要记：**| 新的代码技巧（比如快速查数、清洁数据、分组统计等） | 找最相关的 .ipynb 文件，加在相应的 section 后面 |

- 基础 Python/Pandas 语法（这些 Google 就能查）| 新的坑点或报错解决方案 | 对应 .ipynb 的"🚨 坑点"section，或者 99_常见错误速查.md |

- 一次性的、以后再也不会用的奇葩需求| 新的快速操作（最常用的技巧） | 00_导航与索引.md 的"快速定位表"或"最常用的 3 个操作" |

- 官方文档里有的东西

- 没有实际问题的理论讲解### 怎么写新内容？

### 4.2 加笔记的完整流程

**第 1 步：识别类别**
- 这是什么类型的内容？（数据定位/清洁/融合/分析/高级/错误）
- 应该加到哪个 .ipynb 或 .md 文件？

**第 2 步：找到正确的位置**
- 打开对应的文件
- 找到相关的 section（比如"## 2️⃣ 快速选数"）
- 一般加在该 section 的末尾，或者加在"🚨 坑点"section

**第 3 步：按模板写内容**
- 用上面提供的模板格式
- 确保有"何时用、代码、说明、坑点"这几部分
- 代码必须能直接运行

**第 4 步：检查并保存**
- 检查代码是否有语法错误
- 检查是否需要更新导航表（00_导航与索引.md）
- 保存文件到正确的位置

**第 5 步：如果是新的快速操作**
- 同时更新 `00_导航与索引.md` 里的"快速定位表"

## 📌 Part 7：笔记的内容风格

### 7.1 代码注释的风格

用**大白话**而不是专业术语：

```
# ❌ 不好（太专业，看不懂）
# 运用 loc 进行条件筛选，通过布尔索引实现行的过滤

# ✅ 好（清晰易懂）
# 【快速查询】找所有"价格 > 5000"的行
result = df.loc[df['价格'] > 5000]
```

### 7.2 说明文字的风格

**必须包含：**
- ✅ **何时用** — "我什么时候需要用这个？"
- ✅ **代码** — 完整、能运行的代码
- ✅ **坑点** — "我可能会犯什么错误？"

**尽量避免：**
- ❌ 理论讲解（只讲实用）
- ❌ 官方文档的复制粘贴
- ❌ 太长的说明（简明扼要）

### 7.3 符号的使用

- ✅ = 推荐写法、正确方法
- ❌ = 错误写法、要避免的做法
- 🚨 = 坑点警告、常见错误
- 💡 = 优化建议、更高效的方法
- 🔍 = 查询操作
- ✍️ = 修改操作
- 🔪 = 删除操作
- 📌 = 重要提示

## 📌 Part 10：特殊情况处理

**情况 A：代码太长**
- 只保留核心部分
- 其他部分加注释说明

**情况 B：一个问题有多个解决方案**
- 都记下来
- 标注各自的优缺点
- 推荐最高效的方案

**情况 C：发现笔记里有错误**
- 立即纠正
- 加注释说明"已修复，因为..."

**情况 D：用户问的问题很新、笔记里没有相关内容**
- 先把解答记到笔记里
- 然后再回答用户

## 🎯 总结：AI 的核心任务

1. **主动识别** — 用户说到某个技巧/坑点时，主动提出"要不要记到笔记里？"

3. **保证质量** — 确保代码能运行、说明清楚、格式统一
4. **持续更新** — 发现更高效的方法、发现坑点时及时更新
5. **保持简洁** — 笔记是实用工具，不是教科书
